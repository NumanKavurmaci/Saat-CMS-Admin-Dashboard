# Deployment and operations

This runbook covers deploying the SaatCMS Admin Dashboard to Render, verifying both access modes, rotating credentials, and recovering from a bad release. It describes the repository's current deployment configuration; it does not by itself mean a live dashboard service has been created.

For local work, see [Development](development.md). For symptom-based diagnosis, see [Troubleshooting](troubleshooting.md).

## Production topology

The dashboard and backend are separate Render web services:

```text
User browser
  -> Dashboard web service
       -> public middleware/health request (visitor or account)
       -> CMS request + server-side bearer token (account only)
            -> SaatCMS backend web service
                 -> PostgreSQL
```

The dashboard has no database and stores no user records. Account configuration is copied from the backend's `CMS_API_KEYS`. Dashboard sessions are signed with a separate `DASHBOARD_SESSION_SECRET`.

## Render Blueprint

The root `render.yaml` defines one Node web service:

| Setting | Repository value |
| --- | --- |
| Service name | `saatcms-admin-dashboard` |
| Plan | `free` |
| Build command | `npm ci && npm run build` |
| Start command | `npm run start` |
| Health check | `/login` |
| Node version | `22.15.0` |
| Backend origin | `https://backend-developer-take-home-assignment.onrender.com` |
| Request timeout | `30000` ms |
| CMS credentials | `CMS_API_KEYS`, entered in Render (`sync: false`) |
| Session signing key | Generated by Render when absent (`generateValue: true`) |

Render supplies `PORT`; the production server must listen on that port and on `0.0.0.0`. Do not hardcode a production port.

Important Blueprint behavior:

- Render prompts for `sync: false` values only during initial Blueprint creation.
- Later Blueprint syncs ignore `sync: false`; update `CMS_API_KEYS` in the existing service's Environment settings or a linked environment group.
- `generateValue: true` creates a random 256-bit value only when the variable does not already exist.
- A successful HTTP health check is any `2xx` or `3xx` response. `/login` checks that the dashboard process can serve requests without making the dashboard's health depend on backend availability.

See Render's official [Blueprint specification](https://render.com/docs/blueprint-spec), [web service guide](https://render.com/docs/web-services), and [health check guide](https://render.com/docs/health-checks).

## Release ownership

Treat these as separate responsibilities:

- **Repository:** application code, tests, `package-lock.json`, CI, and `render.yaml`.
- **Render dashboard service:** real secrets, linked branch, region, domains, deploy state, and logs.
- **SaatCMS backend:** endpoint behavior, backend deployment, PostgreSQL readiness, and the authoritative `CMS_API_KEYS` value.

Changing one layer does not automatically update the others. In particular, separate Render services do not share environment variables unless an environment group is deliberately linked to both.

## Before the first deployment

Confirm all of the following:

- The intended release commit is on the branch Render will deploy, normally `main`.
- CI is green for that commit.
- The local production build passes.
- The backend origin is correct for the target environment.
- `CMS_API_KEYS` contains at least one valid `editor` or `admin` entry and matches the backend exactly.
- A disposable editor account, admin account, Content record, Channel, and EPG data plan are available for smoke testing.
- No real secret appears in Git, CI variables, `render.yaml`, documentation, or screenshots.

Recommended pre-release commands:

```powershell
npm ci
npm audit --omit=dev
npm run typecheck
npm run lint
npm run test:coverage
npm run build
```

## First deployment with the Blueprint

1. Push the reviewed release commit to the repository/branch Render will track.
2. In Render, create a new Blueprint and select this repository's root `render.yaml`.
3. Review the proposed service name, plan, region, branch, build command, start command, and health check before applying it.
4. When prompted, enter the real `CMS_API_KEYS`. Do not paste this value into a build log or support message.
5. Allow Render to generate `DASHBOARD_SESSION_SECRET`. Keep it separate from all CMS secrets.
6. Apply the Blueprint and watch the build/deploy Events until the service reports healthy.
7. Record the dashboard URL, deployed commit SHA, deployment time, operator, and smoke-test result in the team's release record.
8. Run the complete smoke matrix below before calling the release ready.

If the Blueprint already exists, sync it for non-secret configuration changes, then manage `CMS_API_KEYS` from the service's Environment settings. Do not expect a new prompt for a `sync: false` variable.

## Post-deploy smoke matrix

Use known disposable data. Prefix created records clearly, for example `smoke-20260713-...`, and remove them at the end. Never test destructive flows against production records you do not own.

### 1. Service boundary

- Open `/login` over HTTPS and confirm it renders without an account session.
- Confirm the response is not an error page and the Render deploy is marked healthy.
- Inspect the deployed commit in Render and confirm it matches the approved release.

### 2. Visitor mode

1. Select **Continue as visitor**.
2. Confirm `/dashboard`, `/system`, `/tools/metadata`, and `/tools/playback` are visible.
3. Confirm Content, Live Channels, and EPG are absent from navigation.
4. Open `/content`, `/channels`, and `/epg` directly. Each must return to `/dashboard` with the CMS-account-required notice.
5. Resolve a known public Content ID in the Metadata tool.
6. Test Playback with a known Content ID, user ID, two-letter country, and each relevant device type (`Mobile`, `SmartTV`, or `Web`). Confirm allowed and blocked decisions are represented correctly.
7. Select **Sign in** and confirm the visitor session ends at `/login`.

Visitor calls must not contain a CMS bearer token. Browser developer tools should show only browser-to-dashboard traffic; the backend call occurs on the dashboard server. Use backend diagnostics, if available and safe, to confirm unauthenticated public calls. The automated API-client regression tests also enforce the missing `Authorization` header.

### 3. Editor account

1. Sign in with a disposable editor actor.
2. Confirm CMS navigation and protected list pages load.
3. Create, inspect, edit, and delete a disposable leaf Content record.
4. Create or edit a disposable Channel if the backend permits it.
5. Create, edit, and delete a disposable EPG Program in a conflict-free time window.
6. Confirm an editor receives the backend's expected result for any admin-only operation.
7. Sign out.

### 4. Admin account

1. Sign in with a disposable admin actor.
2. Repeat one protected read and one mutation.
3. Delete the disposable Channel by typing its exact slug. Confirm its schedule is also disposable because the backend cascade is explicit.
4. Sign out.

### 5. Concurrency and failure behavior

- Open the same Content, Channel, or EPG record in two sessions. Save one, then attempt to save the stale copy. Confirm the second change produces a write-conflict/reload path instead of silently overwriting the first.
- Confirm `/system` distinguishes dashboard availability from backend liveness (`/health`) and database readiness (`/ready`).
- If practical in a non-production environment, temporarily use an invalid backend origin and confirm the UI reports an upstream failure without exposing configuration or secrets.

### Release acceptance

The release is ready when:

- Render marks the dashboard healthy.
- Visitor and account boundaries behave as documented.
- At least one public metadata/playback request and one protected request reach the backend successfully.
- Mutations preserve validation, ETag conflict handling, and destructive confirmation.
- No credentials appear in browser storage, browser responses, application logs, or screenshots.
- All disposable records have been removed.

## Routine operations

### Health and readiness

Use the signals for their intended scope:

| Signal | What it proves | What it does not prove |
| --- | --- | --- |
| Render health check `GET /login` | Dashboard process is reachable and can render the login route | Backend or database is healthy |
| Dashboard `/system` liveness | Backend `GET /health` succeeds | Database is ready |
| Dashboard `/system` readiness | Backend `GET /ready` succeeds | Every CMS mutation will succeed |
| A protected smoke read | Dashboard session, CMS credential, network, and one backend path work together | All write paths are safe |

When `/login` is healthy but `/system` is degraded, investigate the backend before rolling back an otherwise healthy dashboard.

### Logs and request correlation

Start with the dashboard's Render Events and Logs, then correlate with backend logs. Capture:

- timestamp and timezone
- deployed dashboard commit
- route and operation
- HTTP status and stable error code
- `X-Request-Id` shown by the dashboard, when present
- actor role, but not the actor secret

Never capture raw request headers, cookies, login form payloads, `CMS_API_KEYS`, or `DASHBOARD_SESSION_SECRET`.

### Environment drift check

Review configuration after any backend move, credential change, or Blueprint sync:

- `SAATCMS_API_BASE_URL` points to the intended backend origin, without an API suffix.
- The dashboard and backend have matching CMS actors, roles, and secrets.
- `DASHBOARD_SESSION_SECRET` exists only on the dashboard and remains at least 32 characters.
- `SAATCMS_REQUEST_TIMEOUT_MS` remains between 1,000 and 120,000 ms.
- The deployed branch and commit are the intended release.

## Credential rotation

### Rotate a CMS account secret

`CMS_API_KEYS` must stay synchronized across the backend and dashboard. Plan a short maintenance window unless both services use a deliberately shared environment group.

1. Generate a new high-entropy secret. Do not reuse the session signing secret.
2. Record the exact actor ID and role being rotated.
3. Update the backend's `CMS_API_KEYS` first and deploy it.
4. Immediately update the dashboard service's `CMS_API_KEYS` and deploy it.
5. Sign in with the new credential and run one protected read.
6. Confirm the previous credential is rejected.
7. Check both services' logs for authentication errors, without logging either secret.

There can be a brief mismatch between steps 3 and 4. During that window, requests for the rotated actor can fail. Existing dashboard cookies do not contain the bearer secret, but the dashboard resolves the actor against its current environment on every protected request. Removing an actor invalidates that actor's sessions on their next use.

If both services are linked to an environment group containing `CMS_API_KEYS`, update the group once and monitor both resulting deploys. Still verify both services, because deploy completion is not simultaneous.

### Rotate the dashboard session secret

1. Announce that all dashboard users will be signed out.
2. Replace `DASHBOARD_SESSION_SECRET` in the dashboard service only.
3. Deploy the dashboard.
4. Confirm old cookies return users to `/login` and new account/visitor sessions work.
5. Run one public and one protected smoke request.

This rotation does not change backend credentials. Its expected impact is invalidating every existing dashboard session.

### Suspected secret exposure

Treat exposure as an incident, not a documentation cleanup:

1. Rotate the affected CMS or session secret immediately.
2. Redeploy every service that uses it.
3. Verify old credentials are rejected and review access logs for misuse.
4. Remove the secret from the current files and coordinate repository-history cleanup with the repository owner. Do not casually rewrite shared Git history.
5. Document the incident without reproducing the secret.

## Rollback

### Code-only dashboard incident

1. Confirm the backend is healthy and the failure began with a dashboard deploy.
2. In the dashboard service's Render **Events** page, select a recent known-good successful deploy and choose **Rollback**.
3. Confirm `/login`, visitor mode, and one protected read.
4. Review the service's configuration after rollback rather than assuming every external setting moved with the code.
5. Render's dashboard rollback disables autodeploy as a safeguard. Keep it disabled while fixing the cause, then deliberately re-enable it when the forward fix is ready.
6. Create a forward-fix branch and run the full CI/smoke process before redeploying.

Render reuses the selected build artifact for a rollback, while configuration behavior varies by setting. Environment groups are especially separate from a service build. Follow the current [Render rollback documentation](https://render.com/docs/rollbacks) and verify the effective service settings after every rollback.

### Credential/configuration incident

A code rollback alone is not enough when the dashboard and backend credentials drift.

1. Decide which credential set is authoritative.
2. Restore matching `CMS_API_KEYS` values on both services.
3. Deploy/restart as required for both services to use the corrected configuration.
4. Verify new login, one protected read, and rejection of the unwanted credential.
5. Rotate again if either value might have been exposed.

### Backend incident

Do not roll back the dashboard merely because `/system` reports backend liveness/readiness failure. Triage and recover the backend separately. Roll back both services only when the incident analysis shows that their contracts or credentials must move together.

## Incident triage sequence

Use this order to avoid changing the wrong service:

1. Can Render reach dashboard `/login`?
2. Does visitor login create a session?
3. What do backend `/health` and `/ready` show on `/system`?
4. Do public Metadata and Playback calls work?
5. Does an account login succeed?
6. Does one protected read succeed?
7. Is the error isolated to a mutation, role, record, or stale ETag?
8. What changed most recently: code, backend deploy, environment, domain, or data?

Capture evidence before rotating credentials or rolling back, but never capture secret values.

## Monitoring status

Datadog integration is intentionally deferred. The current operational baseline is:

- Render deployment Events and service Logs
- Render HTTP health check against `/login`
- the dashboard `/system` view for backend liveness/readiness
- structured backend error codes, with request IDs on Content mutation errors
- CI quality gates before deployment

There is currently no external uptime probe, centralized log aggregation, distributed tracing, custom metrics, alert policy, or formal SLO in this repository.

If monitoring is added later, start with:

- external HTTPS probes for dashboard `/login`, backend `/health`, and backend `/ready`
- alerting on sustained `5xx`, timeouts, failed deploys, and readiness failure
- dashboard-to-backend latency and error rate, separated by public versus CMS route family
- request-ID propagation for correlation
- secret redaction at ingestion, with no form bodies, cookies, bearer headers, or environment dumps

Monitoring work should be a separate reviewed phase with ownership, retention, privacy, and alert-routing decisions. Do not add a Datadog agent or browser SDK ad hoc.

## Operations checklist

### Every release

- [ ] CI and production audit pass.
- [ ] Approved commit is on the deployed branch.
- [ ] Render Blueprint/configuration diff is reviewed.
- [ ] No secret is present in Git or logs.
- [ ] Render health check passes.
- [ ] Visitor smoke passes.
- [ ] Account smoke passes.
- [ ] Disposable records are removed.
- [ ] Release commit, operator, and outcome are recorded.

### Every credential change

- [ ] Scope and affected actors are recorded without secret values.
- [ ] Backend and dashboard values match.
- [ ] Both services deploy successfully.
- [ ] New login/protected read succeeds.
- [ ] Old credential is rejected.
- [ ] Logs are reviewed for unexpected failures.

### Every rollback

- [ ] Incident is confirmed to be dashboard-specific or cross-service.
- [ ] Known-good deploy is identified.
- [ ] Effective environment/configuration is verified after rollback.
- [ ] Autodeploy state is reviewed.
- [ ] Smoke matrix is rerun.
- [ ] A forward fix and incident note are created.
